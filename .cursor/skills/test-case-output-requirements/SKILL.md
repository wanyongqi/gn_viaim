# Test Case Output Requirements

## Quick Start

当用户要求生成、补充、改写或汇总测试用例时，按以下规则执行：

1. 先读取需求，不明确处先提问，不新增、不曲解需求。
2. 单个需求文件默认输出为 Markdown 列表树结构，不直接使用 `#`、`##`、`###` 标题。
3. 用例名称统一使用 `tc-Px：` 前缀，`pc:` 统一写状态句。
4. 单条用例优先控制在 2 步，最多不超过 3 步，每步都要有对应预期。
5. 优先用分组下沉处理录音类型、平台、型号等维度，不在标题里混杂多个维度。

## Core Format

- 默认结构：`- 功能点名称` -> `- 【一级分组】` -> `- 【二级分组】` -> `- tc-Px：用例标题` -> `- pc:` -> `- 步骤` -> `- 预期结果`
- 标题使用主谓宾表达，尽量描述“什么场景下什么行为结果正确/异常”。
- `pc:` 只写简洁明确的状态，不重复写无意义动作过程。
- 步骤写用户动作，预期写可观察结果。
- 文案统一使用固定术语，如 `toast`、`弹窗`、`横幅`、`kill APP`、`切后台`、`最小化后切回前台`。

## Split And Merge Rules

- 不要强行合并用例。
- 若一条用例同时覆盖两个独立验证目标，优先拆开。
- 录音类型默认分开写，除非用户明确允许合并。
- Android 和 iOS 默认视为功能一致，除非存在系统限制或交互差异。
- 多维度覆盖优先通过分组下沉实现，而不是把多个维度塞进单条标题。

## Priority And Coverage

- `P0`：核心功能正向用例。
- `P1`：常用场景，包含重要错误和边界值。
- `P2`：不常用场景。
- `P3`：异常和边界反向场景。
- 默认关注主流程、异常、边界、中断、恢复、断网、重试、切后台、kill APP、状态恢复等场景。

## File Output Rules

- 单个需求默认输出为独立 `md` 文件，文件名格式：`需求名称_测试用例.md`
- 若需求属于某一版本，默认输出到对应版本目录。
- 若用户要求汇总多个需求，可额外生成汇总文件；汇总文件可使用 Markdown 标题组织各需求。
