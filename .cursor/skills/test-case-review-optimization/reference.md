# 测试用例复查优化参考清单

## 目标

- 在不新增、不曲解需求的前提下，优化测试用例的完整性、可执行性、优先级合理性和表达一致性。
- 优先解决漏测、误分级、表达冗余、逻辑重复等问题。
- 优化过程遵循“先审视结论，再决定是否落文档”的原则，避免边看边改导致判断失真。

## 单个复查

### 核心检查项

- 复查异常场景是否存在缺失。
- 复查边界值场景是否需要补充。
- 复查优先级分布是否需要调整。
- 检查用例标题、前置条件、操作步骤、预期结果是否有可顺手优化的地方。

### 复查重点

- 标题是否准确表达单一验证目标，避免一条用例承载多个核心断言。
- `pc:` 是否是简洁明确的状态句，避免把操作过程写进前置条件。
- 步骤是否过长，是否存在可以收短或拆分的情况。
- 预期结果是否可观察、可判断，避免写成模糊描述。

## 中阶段优化

### 优化方向

- 对异常场景和边界场景进行补漏。
- 对当前优先级进行复查，确认是否符合实际风险和使用频率。
- 统一文案风格，顺手收短冗余步骤。
- 检查功能交互范围是否漏测，尤其关注状态切换、中断恢复、失败重试等场景。

### 重点关注范围

- 主流程是否闭环，关键功能是否覆盖完整。
- 异常、边界、中断、恢复场景是否存在缺口。
- 涉及多录音类型、多平台、多耳机型号时，分组是否清晰、维度是否混杂。
- 相似用例之间是否仅是文案重复，还是确实承载不同验证目标。

## 最终复查

### 收敛原则

- 复查测试用例是否存在逻辑重复，需要删除或合并的地方。
- 仅在验证目标一致的前提下进行合并，不为压缩数量强行合并用例。
- 3 大录音类型用例分开写，无需合并。
- 若一条用例同时覆盖“交互是否正常”和“修改后是否生效”等独立目标，优先拆开。

### 输出要求

- 先给出一轮全量审视与结论，不直接改文件。
- 按“高 / 中 / 低”整理问题清单，便于评审和确认。
- 待确认后，再统一落到正式测试用例文档中。

## 建议补充内容

### 增加优先级复查判定标准

- 明确什么情况应判为 `P0 / P1 / P2 / P3`。
- 增加“用户高频主路径”“高风险异常”“低频边界”等判断示例，减少主观分级偏差。

### 增加通用补漏清单

- 固定检查：异常、边界、中断、恢复、重试、切后台、kill APP、断网、重连、弹窗关闭、权限变化。
- 复查时按清单逐项扫一遍，降低漏测概率。

### 增加拆分与合并判断规则

- 验证目标不同时拆开。
- 前置状态不同时拆开。
- 结果判定不同时拆开。
- 仅在验证目标一致时考虑合并。

### 增加文案统一规则

- 标题统一使用主谓宾表达。
- `pc:` 统一写状态，不写动作链路。
- 步骤统一写用户动作。
- 预期统一写可观察结果，避免使用泛化表述。
