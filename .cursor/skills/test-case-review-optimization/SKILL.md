# Test Case Review Optimization

## Quick Start

当用户要求复查或优化测试用例时，按以下顺序执行：

1. 读取目标测试用例文件。
2. 读取项目中的 `用例输出要求.md` 作为格式和结构基线。
3. 先做一轮全量审视，输出结论，不直接改文件。
4. 审视结果按 `高 / 中 / 低` 分级整理。
5. 用户确认后，或用户明确要求直接修改时，再落到正式文档。

## Review Focus

- 是否漏测异常场景、边界值场景、中断恢复场景。
- 是否存在优先级分布不合理的问题。
- 是否存在标题不清晰、`pc:` 不像状态句、步骤过长、预期不可观察的问题。
- 是否存在逻辑重复、误合并、该拆未拆的问题。
- 录音类需求默认按类型分开写，不强行合并。

## Editing Rules

- 不新增、不曲解需求；不替用户补写未经确认的功能逻辑。
- 标题尽量只承载 1 个核心断言，使用主谓宾表达。
- `pc:` 只写状态，不写动作链路。
- 单条用例优先控制在 2 步，最多不超过 3 步。
- 每个步骤对应一个可观察预期，避免“正常”“无问题”等模糊表述。
- 若一条用例同时覆盖“交互是否正常”和“修改后是否生效”等独立目标，优先拆开。
- 仅在验证目标一致时合并，不为压缩数量强行合并。

## Recommended Workflow

### Phase 1: 全量复查

先输出问题清单，至少覆盖：

- 漏测项
- 优先级问题
- 拆分/合并问题
- 文案表达问题
- 可顺手收短的步骤

### Phase 2: 生成建议

按 `高 / 中 / 低` 输出：

```md
## 复查结论

### 高
- 漏测了：
- 优先级不合理：
- 需要拆分：

### 中
- 文案可优化：
- 步骤可收短：

### 低
- 表达可统一：
- 可选补充项：
```

### Phase 3: 落文档

在用户确认或明确要求直接修改时：

- 先保留原有功能分组思路，再做局部优化。
- 优先修复高风险问题，再处理文案统一。
- 修改后复查是否与 `用例输出要求.md` 冲突。
